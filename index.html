WhatsApp strategy

WhatsApp retained — regional edge broker architecture

The multi-region challenge: At ~1B DPPs/month covering 6 global regions, a single WhatsApp Business API number and webhook endpoint cannot serve factories in Bangladesh, Vietnam, Turkey, and India simultaneously. Regional data sovereignty laws (India DPDP Act, GDPR, PDPA) also prohibit routing one country's factory data through another country's server. The solution is a WhatsApp Regional Edge Broker (WREB) — one per region, each a fully autonomous intake node.
W
WhatsApp Regional Edge Broker (WREB)
One per region — fully autonomous intake node
Each of the 6 global regions gets a dedicated WREB: a containerised microservice (EKS pod, 8 replicas) that owns a region-specific WhatsApp Business number, maintains a local Kafka topic for ingestion, and processes factory documents entirely within its regional boundary before forwarding structured JSON to the global DPP pipeline. The WREB never forwards raw documents cross-region — only extracted, schema-validated JSON fields.

Regional number routing

Each exporter's WhatsApp number is pre-configured at onboarding to their regional WREB number. India factories → +91 VeriPass India number. Bangladesh → +880 VeriPass BD number. Vietnam → +84 VeriPass VN number. Turkey → +90 VeriPass TR number. LLL maintains a GS1-registered DPP submission portal that auto-routes WhatsApp Business API calls to the correct regional WREB via Meta's Cloud API multi-number support.

SME vs enterprise traffic split
WhatsApp for SMEs · API/ERP/IoT for large factories
At planetary scale, ~7.5% of DPPs arrive via WhatsApp (SME factories <200 employees). The remaining ~92.5% arrive via ERP direct API (SAP, Oracle, Dynamics connectors), IoT fabric sensor feeds, and EDI/AS2 B2B integrations at large factories. WhatsApp remains the critical SME channel — covering the long tail of global textile producers who cannot afford ERP systems.
~72M/mo via WhatsApp ~888M/mo via API/ERP
L
Rate limit management
Meta API limits handled by WREB queue
Meta WhatsApp Business API: 1,000 messages/second per phone number. Each WREB handles up to 1,000 msg/sec = 86.4M messages/day per region. Since each DPP averages 8 WhatsApp messages (submit + OCR confirm + field corrections + approval + QR delivery + alerts), one number supports ~10.8M DPPs/day per region — far above any single regional demand. WREB queues outbound messages in Kafka to stay within Meta rate limits.
~10.8M DPPs/day capacity
Complete workaround: WhatsApp → Kafka → Global Pipeline
Step 1 — Regional intake: Factory supervisor sends photo to regional WREB WhatsApp number. WREB webhook (EKS pod) receives the image via Meta Cloud API.
Step 2 — Edge AI processing: WREB invokes the regional IBM Granite 4.0 Micro instance (on the regional GPU cluster) for OCR and field extraction. All processing stays within the regional boundary — no raw document leaves the country.
Step 3 — Structured publish: Extracted JSON (15 ESPR fields, confidence scores, document metadata) is published to the regional Kafka ingestion topic. Only structured, anonymised data flows to the global Kafka mesh.
Step 4 — Global pipeline: The global VeriPass pipeline (EKS, Cassandra, Besu, CIRPASS) processes the structured record identically regardless of whether it arrived via WhatsApp, SAP ERP, or IoT sensor.
Result: WhatsApp is fully retained as the SME intake channel. Factory supervisors experience no change. The WREB abstraction layer makes WhatsApp behave identically to an ERP API at the pipeline level — it is just another Kafka producer.
